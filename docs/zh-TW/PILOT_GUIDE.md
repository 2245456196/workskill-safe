# Pilot 試行指南

## Pilot 是什麼

Pilot 是一個受控、小規模的實際測試場景。它不是單純跑 Demo，也不是把大量真實資料丟進工具，而是選一個範圍清楚的工作流程，驗證它能否安全地轉換成可重複使用的 Agent Skill。

## Pilot 要回答的問題

1. 工具能否發現需要人工確認的內容？
2. 去識別化後，是否仍保留足夠的流程資訊？
3. 產生的 Skill 是否能套用到第二個不同案例？
4. 哪些規則造成誤判或漏判？
5. 哪些程式、文件或檢查流程需要修正？

## 第一個建議場景

建議先做「會議紀錄轉行動清單」。

原因：

- 流程結構清楚，容易驗證輸入與輸出。
- 可明確區分決策、任務、負責人、期限、阻塞與風險。
- 能用完全虛構案例做第二次驗證。
- 相較於提案或商業計畫，初次測試的機密風險較低。

## 執行方式

1. 在私人本機資料夾複製一份小型工作筆記。
2. 先人工移除明顯的公司、客戶、專案、姓名與連結。
3. 執行 `scan`。
4. 執行 `sanitize`，查看遮罩後內容與稽核報告。
5. 進行人工複核，確認無法從上下文推回真實對象。
6. 執行 `scaffold --reviewed` 生成 Skill 草稿。
7. 執行 `validate`。
8. 用另一份完全虛構的會議紀錄測試該 Skill。
9. 只記錄聚合結果與需要修正的規則，不公開原始內容。

## 建議記錄欄位

- Pilot 編號
- 測試流程類型
- 輸入規模
- 偵測項目數
- 人工發現的漏判數
- 誤判數
- 是否成功生成 Skill
- 第二案例是否可用
- 需要新增或修改的規則
- 最終決策：保留、修改或停止

## 成功標準

- 原始內容沒有進入公開 repo。
- 去識別化後仍能保留可重複的方法。
- 生成的 Skill 能處理第二個虛構案例。
- 至少發現一項可改善的程式、規則或文件問題。
